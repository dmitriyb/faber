# CLI — entry points, dispatch, logging

## What it is

The `faber` binary's user surface: cobra-based subcommand dispatch, exit
codes, and structured logging initialization. Thin by design — every
subcommand is a composition of module entry points, and nothing below the
CLI reads flags or writes to the terminal directly.

## Library choice: cobra

`faber` dispatches on [`cobra`](https://github.com/spf13/cobra) (pinned to the
same version spexmachina uses), matching how spexmachina's own `spex` CLI is
built (`spex`'s `spec/cli/`, `arch_root_command.md`). It provides declarative
subcommand registration, auto-generated `--help` at every level, and POSIX
flag parsing via `pflag`, replacing what used to be a hand-rolled `flag`-based
`switch` plus custom `wantsHelp`/`printUsage` shims that only covered `run`
and `resume` — `faber --help` used to fall through to the unknown-command
path (`unknown command "--help"`, exit 2, stderr) because root-level help was
never wired. This is an intentional exception to the "Go standard library
first" constraint, for the same reason spexmachina's CLI-infrastructure
proposal gives it: reimplementing subcommand registration, help formatting,
and flag parsing adds complexity without value, and it is the largest
dependency in the binary for a structural gain the hand-rolled version could
never close on its own (uniform help at every level).

Two departures from spexmachina's `cli`/`cmd/spex` split, both load-bearing
for faber specifically:

- **The whole command tree stays inside `config`**, not a separate `cli`
  package. faber's CLI has always lived in `config` so `config/cli_test.go`
  can drive it in-process — see "No hidden state" below; spexmachina's split
  has no equivalent cross-module dependency-injection seam to preserve.
- **cobra's autogenerated `completion` subcommand is disabled**
  (`CompletionOptions.DisableDefaultCmd`). spexmachina keeps it; faber's
  command set is a fixed, documented surface (below) and `completion` is not
  part of it.

## Subcommand registration pattern

One file per subcommand under `config/` (`cmd_validate.go`, `cmd_build.go`,
`cmd_run.go`, `cmd_resume.go`, `cmd_upgradecheck.go`, `cmd_addkey.go`,
`cmd_listkeys.go`, plus `version.go`), each exporting a
`newXxxCmd(deps Deps) *cobra.Command` that registers its own flags and a
`runXxxE(cmd *cobra.Command, ..., deps Deps) error` body — mirroring
spexmachina's `cmd/spex/*.go` file-per-command split (`newXxxCmd()` +
`runXxxE(cmd, args)`). The one mechanical departure from that template:
`newXxxCmd` and `runXxxE` both take an explicit `deps Deps` parameter
(closed over by the `RunE` closure), because faber's subcommands dispatch
into not-yet-wired module seams (`PackageProver`, `ImageBuilder`, `Executor`,
`JournalStore`, `RunAuditor`, `RegistryController`) that spexmachina's
commands have no equivalent of.

`NewRootCmd(deps Deps) *cobra.Command` wires every subcommand onto the root
via `AddCommand`, exactly like spexmachina's `main.go`. `RunWithDeps` builds
the root, calls `SetArgs`/`SetOut`/`SetErr`, calls `Execute()`, and maps the
returned error to an exit code (see "Exit codes" below) — the same three
setters spexmachina's `cli/root_test.go` uses to test its root command
in-process, which is what keeps `config/cli_test.go` a pure in-process
harness with no subprocess spawned.

Unlike spexmachina's single persistent `--spec-dir` flag (inherited by every
subcommand via `PersistentFlags()`), faber has no root-level persistent
flags. `--config`/`--log-level`/`--log-format` are registered per-command by
two small helpers, `addCommonFlags` and `addLogFlags` — because `add-key` and
`list-keys` deliberately take the narrower set (no `--config`; see below).
Making these persistent on root would silently widen that surface for every
subcommand.

## Subcommands

| Command | Composition | Exit 0 means |
|---------|-------------|--------------|
| `faber validate [--config path] [--emit-ir] [--workflow name]` | Load -> Validate -> Desugar -> WiringChecker -> infra package-resolution proof | the named (or every) workflow would start |
| `faber build [--config path] [--template name]` | Load -> Validate -> ImageBuilder per template | images built and tagged |
| `faber run <workflow> [--param k=v ...] [--config path] [--max-parallel n] [--budget u=n] [--metering path] [--report-json path\|-]` | validate pipeline -> executor with journal, meter, bindings | run settled with every step ok or skipped-by-condition |
| `faber resume <run-id> [--fresh] [--interactive <step-id>] [--report-json path\|-]` | journal load -> version/drift guards -> recovery mode dispatch (failure module) | as `run` |
| `faber upgrade-check [--force]` | run audit (failure module) -> refuse while live/unfinished runs exist | safe to swap the faber binary |
| `faber add-key --role <name> --fingerprint SHA256:… [--comment <c>] [--force]` | security.RoleRegistry load -> AddKey -> atomic save | the role points at the fingerprint (upsert or verified no-op) |
| `faber list-keys` | security.RoleRegistry load -> print | the registry was read and printed |
| `faber version` (also `--version` / `-v` on the root) | print version, commit, build date | — |

`add-key` and `list-keys` manage the global `role → fingerprint` registry
(`roles.json` under faber's config home) the identity binding resolves
against; they take **none** of the shared config flags below (they touch no
`orchestrator.yaml`), only `--log-level`/`--log-format`. faber never writes
key material — only the fingerprint plus an optional label. The subcommands
are thin dispatch; the store, validation, atomic write, idempotency, and
`--force` refusal live in the security module (`spec/security/arch_role_registry.md`,
`impl_role_registry.md`). A malformed `--fingerprint` or `--role`, or a
missing required flag, is a usage error (exit 2); a refusal to re-point an
existing role without `--force`, or an IO error, is exit 1.

`faber --help`, `faber -h`, `faber help`, and every subcommand's `-h`/`--help`
(e.g. `faber run --help`, `faber help run`) print usage and flag defaults to
stdout and exit 0 — never the exit-2 usage error, and never only reachable at
the subcommand level. `--report-json` writes the machine-readable run report
to a path (`-` = stdout, after the human report).

Flags shared by validate/build/run/resume: `--config` (default `orchestrator.yaml`), `--log-level`
(default `info`), `--log-format` (auto/json/text). `--config` names the **root
project file**; `Load` transitively pulls its `include:` closure and merges the
component libraries before validation (see `arch_loader.md`). A single-file config
with no `include:` behaves exactly as before, so the default and every existing
invocation are unchanged.

## Exit codes

0 success; 1 validation or run failure (details already reported); 2 usage
error. `validate` reports *all* errors before exiting — the multi-error
discipline of the Loader and WiringChecker surfaces here as one combined,
field-path-sorted listing.

This contract predates cobra and is preserved exactly, not relaxed to
cobra's own defaults (uncaught error → stderr, exit 1, no usage-vs-operational
distinction). Every subcommand's `RunE` returns a plain `error` for the exit-1
cases and a `usageErr(err)`-wrapped one for the exit-2 cases; a small
`cliError{code, err}` implementing `ExitCode() int` carries the code back
through `RunWithDeps`'s single `errors.As` check at the top — the same
convention spexmachina's own `cmd/spex/*.go` commands use for their exit
codes (`emitError`, `ingestError`, `diffError`, each with an `ExitCode() int`
method that `main.go` inspects). `RegistryUsageError` (the existing
security-seam usage-error type; see `add-key` above) carries the same
`ExitCode() int` method rather than a type-switch at the call site. cobra's
own flag-parsing errors are routed through the same mechanism via
`root.SetFlagErrorFunc`, so a malformed flag still exits 2 exactly as it did
under the old `flag.ContinueOnError` FlagSets.

## Logging

`slog` initialized once at startup: JSON handler when stderr is not a TTY, text
handler when it is (overridable via `--log-format`). Each module receives a child
logger via `logger.With("component", name)` — no global logger, no
`slog.SetDefault` (conductor's D3, carried over). Secrets never reach the logger:
resolver outputs and credential material are typed so they cannot be passed to
logging calls (dedicated opaque types with redacted `String()`).

## Design rationale

- **cobra for dispatch, with the exit-code contract pinned rather than
  adopted from cobra's defaults.** See "Exit codes" above — a released tool's
  exit codes are part of its contract with scripts that call it; the
  migration to cobra changes the dispatch machinery and the help surface, not
  that contract.
- **`run` embeds `validate`.** There is no code path that executes an IR that did
  not just pass the full validation pipeline in the same process. `--emit-ir`
  exists so the validated artifact is inspectable and diffable, not so it can be
  fed back in.
- **No hidden state.** The CLI passes `*Config`, the IR, and constructed module
  values down explicitly; nothing reads globals. This is what keeps every module
  testable without a CLI harness — and, per "Subcommand registration pattern"
  above, is why the whole cobra tree stays inside `config` rather than moving
  to a separate `cli` package the way spexmachina's does.
