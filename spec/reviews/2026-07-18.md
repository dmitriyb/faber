# Faber — Combined Review & Remediation Plan (2026-07-18)

## 0. About this document

Consolidated remediation reference for faber. It merges two review passes — **Pass 1 (ledger)**
(input validation / robustness, failure-mode correctness, operator ergonomics, spec↔code fidelity)
and **Pass 2 (orthogonal)** (trust boundaries & untrusted-input handling, concurrency & races,
invariants / property & fuzz testing, on-disk format evolution & migration).

Every finding was verified against source (`file:line` anchors exact against `main` at writing
time). Framing is neutral engineering throughout. **Remediation scope = the union of both passes.**

**How to read a finding:** `ID — title` · `file:line` · **Impact** · **Direction** · *Disposition*.
**Severity:** P1 = fix first · P2 = should fix · P3 = minor.
**Disposition:** `fix` · `spec-only` (code correct, spec stale) · `strong-point` (do not regress).

**Work order** (per the agreed chain — spec: change→review→fix, then code: implement→review→fix,
then an independent conformance pass): the group numbering is a **label, not a march order**. The
only hard sequencing is data-dependency — the run-level lock (F2 / CC-F1) lands before the resume /
update guards that rely on it, and the handoff version field (FE-F3) lands with the F6 re-keying;
everything else is independent. Group by code surface; do not re-touch the same function once per
finding.

Faber's guiding principle is unchanged and must be preserved: **faber is mechanism, not policy.**
Any change that teaches faber a domain word (a tracker, a gate service, a spec tool) is wrong by
construction — find the config seam. Every real control is enforced from outside the container;
the thing inside the box is untrusted; no secret is ever materialized inside a box.

## 1. Design direction adopted — on-disk format versioning (Tier 1 + guards)

The only cross-cutting redesign this pass. (Portitor carries the larger L1/L2 redesigns; faber's
findings are otherwise localized fixes.)

- Add a **format-version stamp** to each persistent artifact that lacks one: the journal header,
  the handoff record, the faber↔faber-box result contract, and the resolved image tag/pin in the
  resume path. (`orchestrator.yaml` already carries `version: 1`; the IR already carries
  `IRVersion`.)
- The schema version is **independent of the application version** — bumped only when the on-disk
  format actually changes, so guards trip only on a real format change, not on every release.
- **Guards, not migrations:**
  - **upgrade-guard (read-only pre-upgrade check).** A new `faber upgrade-check` subcommand reports
    and refuses (non-zero) when incomplete runs exist (a run with no terminal state in its journal),
    listing them, with `--force` / mark-abandoned to override. It does **not** update faber — the
    binary swap is external (rebuild / release); this is the pre-flight an operator or deploy step
    runs first, encoding the rule "faber is not upgraded mid-run". It composes with the run-level
    lock (F2 / CC-F1), which already knows which runs are live.
  - **resume / destructive-guard.** A newer binary meeting an older-schema journal or config makes
    `resume` / `--fresh` / GC **fail closed** with a clear message ("journaled by schema vX, current
    vY; no auto-migration — finish on the old version or start `--fresh`"), never silent
    misinterpretation.
- **Explicit consequence (no surprise):** across a schema bump, in-flight runs are finished on the
  old binary or restarted `--fresh`. There is **no auto-migration** — that is deferred Tier 2 (§3).
- **faber↔faber-box contract:** the host stamps a contract version into the box env; the box
  refuses / warns on mismatch and stamps its own version into `result.json`, which the host asserts
  on extract. (faber-box ships from the host as the same build, so a mismatch is a
  `FABER_BOX_BIN`-misconfiguration detector, not a routine-migration path.)

### Appendix findings — strong points (do NOT regress)

- Journal discipline: synced single-line appends under a mutex, torn-tail repair on reopen,
  `O_EXCL` run creation, last-wins replay; symmetric line-size bound (append refusal + scanner
  buffer, `sc.Err()` checked) so oversized records fail at write time.
- Event-loop ownership: all graph state mutated on the run goroutine; one event per worker;
  `send` selects on `done`; no double-close / double-send / goroutine leak.
- `result.json` / handoff writes are atomic temp+rename inside the mounted dir — no torn read.
- Skip/annotation encodings are forgery-resistant (empty `InputHash` + reserved reason;
  `Attempt == 0`; box reasons namespaced `box:`), so a container record cannot spoof a skip/cache
  annotation or flip the exit code.
- `InputHash` canonicalization is rigorous (sorted keys at every depth, `json.Number` verbatim,
  RawMessage trailing-data rejected).
- Teardown / cancellation run on detached contexts; `BindingSet` is stateless per step; secrets
  redacted at the type level with a single `reveal()` caller.
- Admitter lock discipline: estimates gathered outside the ledger lock; reservation/spent mutation
  under one mutex with defensive double-release.
- Docker query adapter never scrapes free text (`--format '{{json .Field}}'` + typed decode).

## 3. Deferred / backlog (retained, intentionally out of this pass)

- **On-disk migrations / backward-compat readers** (Tier 2 of §1) — no auto-migration across a
  schema bump; deferred until the upgrade model demands it.
- **`branch-delete`** side-effect handling in `release-bead` once portitor grows the verb.
- Any richer cross-version reconciliation of metering cost records (unit/currency drift) beyond
  the Tier-1 stamp.

**Pre-ship verification (workstream verify step, not feature backlog):**
- Race-detector run of the scheduler/journal/admitter under `-race` with a randomized-DAG harness.
- The `realinfra`-tagged integration suite (docker + nix) for the image-tag atomicity (CC-F3) and
  preflight (L-P2c) fixes.

## 5. Fuzz & property-test plan

No native fuzz targets exist. Highest value (functions verified present):

1. `FuzzJournalLoad` — `failure.Load`; properties: no panic; prior `Results` preserved across
   repair+append; `Load∘write∘Load` fixpoint; unknown-status vs torn-tail discrimination (IN-F4).
2. `FuzzExtractAdapt` — write bytes as `result.json`, run `ExtractResult` + `adaptResult`; assert
   `out.Validate()==nil` for every input and no panic; handoff-pointer stays under the result dir
   (TB-F1); usage clamped (TB-F3).
3. `FuzzParseRef` / `FuzzRewriteStepRefs` (`config/refs.go`, `celref.go`) — totality (literal XOR
   one-ref XOR error); idempotence; the agreement property that any `steps` token outside a string
   literal is fully rewritten or rejected (drives IN-F1).
4. `FuzzConfigAssemble` — YAML → Load → Validate → Desugar → CheckWiring → HashIR; no panic, no
   hang (alias-bomb → error not hang), determinism, field-pathed errors; step-id collision seeds
   (IN-F2), `loop.max` bound.
5. `FuzzParseItems` + splice rollback invariance (`generate.go`); item-id charset seeds (IN-F6);
   the node ceiling (IN-F3).
6. `FuzzInputHash` — equal values ⇒ equal hash regardless of map iteration; RawMessage trailing
   bytes rejected; deep-nesting no overflow.
7. Property test (deterministic rand, `-race`): random DAGs run to settlement with a watchdog →
   every node terminal, journal count == settlements; then resume with a BoxRunner that fails if
   called → all agent nodes journal hits, report totals identical. Targets the evDeferred/wakeParked
   race window (CC-F2) and the run-level-lock guarantees (CC-F1).
